# v622 同期コーディネータ統合

## 目的
共有フォルダ運用で並行して発生していた起動時同期、定期同期、保存後更新、手動同期を直列化し、古い一覧取得結果が新しい保存結果を上書きする競合と、SMB/共有フォルダへの重複I/Oを抑制する。

## 実装
`src/renderer/src/main.tsx` に優先度付き同期キューを追加した。

優先順位：

1. `save` — ページ・DB保存後の軽量更新
2. `manual` — 手動再読み込み、管理同期
3. `startup` — 初期キャッシュ表示、遅延した共有フォルダ取込
4. `periodic` — 45秒ごとの軽量同期

### 動作ルール

- すべてのワークスペース更新は `enqueueWorkspaceRefresh()` 経由で直列実行する。
- 同じ優先度かつ共有取込の有無が同じ依頼は、取得対象を合成して1回にまとめる。
- 定期同期は、実行中または待機中の更新がある場合は積まず、次周期へ見送る。
- 保存後更新と共有フォルダ取込は混在させない。保存後に意図せず重い共有フォルダ取込を実行しない。
- 起動時はローカルSQLite由来の一覧を先に表示し、共有フォルダ取込は従来どおりアイドル時へ遅延する。

## 期待される効果

- 保存直後に定期同期が重なって画面状態が巻き戻る可能性を低減。
- タブを開いた直後・初回起動時に共有フォルダ読込が重なりにくくなる。
- 45秒周期の同期が低速共有フォルダで連続キュー化しない。
- DB、ページ、Semantic Indexの保存後更新は低遅延のまま維持。

## 検証

- `src/renderer/src/main.tsx` をesbuildで構文変換し成功。
- `refreshWorkspace()` の直接呼び出しは、キュー処理内部のみへ集約したことを確認。
- プロジェクト全体の `npm run typecheck` は、添付アーカイブに依存関係が含まれないため実施不可。実環境で実行すること。

## 次の推奨

1. 同期キューの待機件数・最終成功時刻・最終エラーを診断パネルへ表示する。
2. DB行PATCH／ページ保存／共有取込を対象に、保存直後の外部同期競合を再現する回帰テストを追加する。
3. 分析ノートのDuckDB同期は別キューに分離し、通常のページ／DB一覧同期を待たせない。
