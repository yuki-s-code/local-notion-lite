# DB行・本文・関連Index 全体監査（v600）

## 結論

ページ本文とDB行本文のIndex経路には差がありました。今回、通常のDB行本文保存をページに近い差分更新へ寄せ、DB行本文のタスクIndexと画面遷移も追加しました。

## 今回修正した経路

### 1. DB行本文のリンクIndex

**修正前**

`upsertDatabaseRowLinkIndex()` はDB行本文の保存ごとに次を実行していました。

```sql
SELECT id FROM pages WHERE trashed = 0
```

そのため、DB行にリンクを1件追加しただけでも、全ページIDをメモリへ読み込んでいました。

**修正後**

本文から抽出した `@リンク` / `local-page://` / `page:` の参照先と、DB行の構造的な子ページだけを `id IN (...)` で照会します。子ページは `parent_id = database-row:<databaseId>:<rowId>` も確認するため、本文中のリンク文字列がなくてもバックリンクIndexを維持します。

### 2. DB行プロパティのIndex

**修正前**

`saveDatabaseCore()` から `saveDatabaseFile()` を通る通常のセル編集は、毎回DBの全行についてJSONハッシュを作り、既存Indexと比較していました。書込みは差分でも、判定に全行走査と全セルJSONの文字列化が必要でした。

**修正後**

通常の画面保存では、Rendererが更新する `row.updatedAt` を使い、変更行ID・削除行ID・並び順変更だけを計算します。

- 通常のセル変更: 対象1行の `database_row_index` / `database_row_property_index` / `database_row_fts` / ハッシュIndexのみ更新
- 新規行: 新規行だけIndex投入
- 行削除: 削除行だけIndex削除
- 行並び替え: 行順だけ更新
- プロパティ構成変更・Index破損・外部取込: 従来どおり安全な全再構築へフォールバック

これにより、大規模DBでセルを編集するたびに全行のセルJSONをハッシュ化する経路を避けます。

### 3. DB行本文のタスクIndex

**修正前**

DB行本文に `- [ ]` を書いても、リンクIndexは更新されますが、`task_index` には入りませんでした。ページ・Journal・Inboxだけが対象でした。

**修正後**

- DB行本文保存時に `task_index` を差分更新
- 初回移行時に既存のDB行本文も一度だけ再構築
- タスク一覧、ホーム画面からDB行本文へ遷移
- タスクの完了状態・期限変更をDB行本文へ書き戻し

移行キーを `task_index_built_v600` に変更しています。旧Indexがある環境でも、最初のタスク一覧取得時に一度だけ再構築されます。

## ページとDB行の現在の差異

| 項目 | ページ本文 | DB行本文 |
|---|---|---|
| リンクIndex | 差分更新 | 差分更新（今回統一） |
| 壊れたページリンクIndex | あり | なし |
| 本文検索Index | `page_search_index` / FTS | DBプロパティFTSはあり、本文専用FTSは未実装 |
| タスクIndex | あり | あり（今回追加） |
| Semantic Index | 対象 | 既に対象 |
| 保存ごとのEmbedding | しない | しない |

## 未実装だが意図的に残した点

### DB行本文専用の全文検索Index

`database_row_fts` は現在、表のプロパティ値検索に使われています。DB行本文まで同じIndexへ混ぜると、表のフィルタ・ソート・検索結果の意味が変わるため、今回は混在させていません。

必要になれば、`database_row_content_search_index` と専用FTSを追加し、グローバル検索・AI検索だけが参照する構造にするのが安全です。

### Semantic Indexの即時再生成

Semantic Indexには既にDB行本文・プロパティが入っています。ただし、保存ごとにEmbeddingを生成すると編集が遅くなるため、現在の「保存時は差分候補を残し、バックグラウンド/明示実行でEmbedding更新」という方針を維持しています。

## タブ導入後の初回起動との関係

Tab復元自体は小さなlocalStorage状態の復元です。初回起動を重くしやすいのは、以下です。

1. 共有フォルダの取込
2. 派生Indexの全再構築
3. DB行子ページ・リンクIndexの全走査
4. 大量DBを最初に開いた場合の表レンダリング

今回のDB行Index修正は、通常編集時の負荷を下げるものです。初回起動については、前回のTab起動監査で入れた「初期ツリーで全DB行本文を読まない」「統合リンクIndex再構築を待機条件から外す」修正と組み合わせて効きます。
