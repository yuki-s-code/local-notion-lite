# V429 — 初回ページ表示・関連情報の性能改善

## 修正した原因

- 関連情報の取得時に、対象ページを特定するためだけに `collectWorkspaceSemanticChunks()` を実行していました。
  これは全ページ、FAQ、データベース行、ジャーナルを再読込するため、ページを開く操作に不適切な負荷でした。
- Semantic Indexサービスが呼出しごとに生成され、ローカルSQLiteから全Embeddingを復元していました。
- 関連情報の文字・タグ再順位付けを、全チャンクに対して実行していました。
- バックリンク索引が空の場合、ページ表示中にワークスペース全体の索引を再構築していました。

## 改善内容

1. Semantic Index内の対象チャンクを直接参照します。ページ表示中の全ワークスペース走査を廃止しました。
2. Semantic Indexサービスと水和済みIndexをVaultService内で再利用します。連続ページ表示ではSQLite読込・JSON復元を繰り返しません。
3. 関連情報では、まず意味類似の上位候補だけを選び、その候補にだけ文字・タグ再順位付けを行います。
4. 関連情報の自動読込を本文描画後（通常850ms、コンパクト250ms）へ遅延しました。手動更新は即時です。
5. バックリンク索引が未作成でも、ページ表示中に全体再構築を実行しません。明示的な索引再構築へ分離しました。

## 運用上の注意

- 新規・更新直後のページは、Semantic Index差分更新が済むまで関連情報に出ません。
- 初回のSemantic Index読込は依然必要ですが、同じアプリ起動中の2ページ目以降はメモリ上のIndexを再利用します。
